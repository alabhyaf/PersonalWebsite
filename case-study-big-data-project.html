Big Data Analytics Web Service

A high performance and reliable web service for analyzing ~1 TB Twitter data

The development was divided in to 3 phases. The first phase was for Q1, Q2. The second phase was for Q3 and the third phase for Q4.

Phase 1

A High Performance Web Service for Data Retrieval

  • Build a performant and reliable web service on the cloud within a specified budget.
  • Designed, developed, deployed and optimizeed functional web-servers that can handle a high load (~ tens of thousands of requests per second).
  • Implemented Extract, Transform and Load (ETL) on a large data set (~ 1 TB) and load the data into MySQL and HBase systems.
  • Design schema as well as configure and optimize MySQL and HBase databases to deal with scale and improve throughput.
  • Explored methods to identify the potential bottlenecks in a cloud-based web service and methods to improve system performance.

Tips while improving performance:

  • The key to improve performance in HBase is to split the tables equally among different region servers. This can be done using:
    create 'test_table', 'f1', SPLITS=> ['a', 'b', 'c']
    OR accessing the hbase gui on amazon.
  • In MySQL database you can also shard the database horizontally to achieve the greater performance boost. (We got a performance boost of 35-40%).
  • Another way to increase performance is increse the number of worker threads and IO threads in undertow. The golden ratio b/w IO threads and worker thread you would want to keep is 1:8.

Lastly, write your own cache. Its fun and would boost your performance off the charts if written properly. Use LRU or LFU or whatever suits your needs. (We used LFU for our application.)

public class LFUCache { HashMap vals; HashMap counts; HashMap> lists; int cap; int min = -1; public LFUCache(int capacity) { cap = capacity; vals = new HashMap<>(); counts = new HashMap<>(); lists = new HashMap<>(); lists.put(1, new LinkedHashSet<>()); } public int get(int key) { if(!vals.containsKey(key)) return -1; int count = counts.get(key); counts.put(key, count+1); lists.get(count).remove(key); if(count==min && lists.get(count).size()==0) min++; if(!lists.containsKey(count+1)) lists.put(count+1, new LinkedHashSet<>()); lists.get(count+1).add(key); return vals.get(key); } public void set(int key, int value) { if(cap<=0) return; if(vals.containsKey(key)) { vals.put(key, value); get(key); return; } if(vals.size() >= cap) { int evit = lists.get(min).iterator().next(); lists.get(min).remove(evit); vals.remove(evit); } vals.put(key, value); counts.put(key, 1); min = 1; lists.get(1).add(key); } }

We were getting a RPS of 20,000 with 7 m4.large instances with a load balancer and a cache on AWS.

Step 4: Q2

Query 2 was about returning a slice of records from the data rather than just a record based on user-id and time. The key to remember here was how to define the key in SQL database and how to design schema the HBase database.

In this query we try to find the topic score of each word in the particular slice that is requested to find top topic words using TF-IDF. Topic words are the words which are most talked about. Lastly, we return user-id and tweet-id of the user's with the highest impact score containing one of the topic words. We also have a number which defines how many topic words to include for each response.

Request Format:

http://dns.of.instance/q3?uid_start=2317544238&uid_end=2319391795 &time_start=1402126179&time_end=1484854251&n1=10&n2=8

time_start: in Unix time / Epoch time format, which marks the starting time search boundary e.g. time_start=1480000000 time_end: in Unix time / Epoch time format, which marks the ending time search boundary e.g. time_end=1490000000 uid_start: marks the starting user search boundary e.g. uid_start=492500000 uid_end: marks the ending user search boundary e.g. uid_end=492600000 n1: the maximum number of topic words that should be included in the response e.g. n1=10 n2: the maximum number of tweets in all that should be included in the response e.g. n2=50

Response Format:

[YOUR_TEAM_NAME],[YOUR_TEAM_AWS_ID] word1:score1 word2:score2... wordn1:scoren1 impactscore1 tid1 text1 impactscore2 tid2 text2 ... impactscoren2 tidn2 textn2

Impact score for each tweet is calculated using retweets, followers, favorite and effective word count.

To keep the MySQL database queries efficient we used a composite key on time and user id as we had to retrieve slices of data. To keep the HBase database queries efficient we split the tables manually by defining keys to split at.

Since this was ranged queries our cache became complicated and we ditched it. But sharding MySQL came in handy for improving the performance. Sharding the database means to put a hashing logic at the application layerto redirect the requests.

Our architecture was still the same as Q2 on the cloud with the difference that SQL database was sharded and replicated. We were able to get an RPS of 2800 on 7 M4.large instances with a LB.

Phase 2 (Q3)

A High Performance Interactive Server (Consistent Read/Write)

In this query, we build a web service that supports READ, WRITE, SET and DELETE requests on tweets. The server was receiving a mix of read, write, set and delete requests to test for correctness and performance.

Request Format

GET /q4?op=&field=&tid1=&tid2=&payload=&uuid=&seq=

op: read/write/set/update field: field to wite,set,or update tid1: the start tweet id tid2: the end tweet id payload: value to update, set uuid: the unique identifier of this sequence of operations seq: the sequence number of this operation

Operations under the same uuid should be handled in the order of the corresponding value of seq. That is, all operations with sequence numbers s or higher need to wait for the operation with the sequence number s-1 to finish. However, operations with different value of uuid represents operations in different transactions, and they do not share the sequence numbers.

Response would be the range read or a success message for operations of set, update and delete. We only used MySQL to implement this query. The most efficient way to achieve high performance was a distributed hash. We would replicate the database on all the instance initially and then would redirect request using the uuid of requests to instances. This would mean all requests of a particular uuid would always go to the same instance , thus maintaining context. A hashing logic was required in the application layer.

To maintain the order of operations we used a Priority Queue in a Hash Map so that the order of operations is not corrupted for each uuid in the application layer. Also, we would generate a response for update,set and delete immediately and would run a thread on the server for the performing the task.

We were able to achieve an RPS of 10,000 on 7 M4.large instances.

Phase 3(Q4)

Shortest path in Graph Database

We had to find the shortest between 2 users using their mentions. For example if user 1 mentioned user2, user3, user4 and user2 mentioned user6,user7. So distance between user1 and user7 is user1 -> user2 -> user7.

Request Format

GET /q5?u1=UID1&u2=UID2

Response Format

TEAMID,TEAM_AWS_ACCOUNT_ID\n UID1-->TID1-->UID2-->TID2-->UID3

We did this in a MySQL database. We had the schema where we had 1 databass, one as with tweet-id as the primary key other columns as the users mentioned. We would do a 2 way BFS from start and end on this database.

The code for the 2 way BFS is as:

public class Solution { public int ladderLength(String beginWord, String endWord, List wordList) { Set beginSet = new HashSet<>(); Set endSet = new HashSet<>(); Set visited = new HashSet<>(); Set wordSet = new HashSet<>(wordList); if(!wordSet.contains(endWord)) return 0; beginSet.add(beginWord); endSet.add(endWord); int len = 1; while(!beginSet.isEmpty() && !endSet.isEmpty()){ // System.out.println(beginSet); // System.out.println(endSet); if(beginSet.size()>endSet.size()){ Set temp = beginSet; beginSet = endSet; endSet = temp; } Set tempSet = new HashSet<>(); for(String word : beginSet){ char[] arr = word.toCharArray(); for(int i =0 ; i < arr.length; i++){ for(char let = 'a'; let<='z'; let++){ char old = arr[i]; arr[i] = let; String target = String.valueOf(arr); if(endSet.contains(target)){ return len+1; } if(!visited.contains(target) && wordSet.contains(target)){ visited.add(target); tempSet.add(target); } arr[i] = old; } } } beginSet = tempSet; //System.out.println(tempSet); len++; } return 0; } }
